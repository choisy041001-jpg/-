## 1. 제품 개요

### 1.1 제품 비전

> "서비스 기획자와 팀원 누구나 단일 HTML 파일 하나를 열기만 하면 — 서버도 설치도 없이 — 화면설계서, 프로토타입, 기획 문서를 즉시 탐색할 수 있는 **Zero-Setup 기획 산출물 뷰어**"

### 1.2 핵심 가치 제안

| 가치 | 설명 |
|---|---|
| **Zero-Setup** | 서버 배포·설치 없이 HTML 파일 하나를 브라우저에서 열면 즉시 실행 |
| **통합 관리** | 분산된 MD 문서·HTML 프로토타입을 단일 파일로 패키징하여 일원화 |
| **즉시 탐색** | 화면 ID 기반 즉시 탐색으로 명세 + 프로토타입 동시 확인 |
| **간편 공유** | 파일 하나를 메신저·드라이브로 전달하면 팀원 누구나 즉시 열람 가능 |
| **오프라인 동작** | 외부 CDN 없이 완전 오프라인 환경에서도 정상 동작 |

### 1.3 성공 지표 (KPI)

| 지표 | 목표 (출시 후 3개월) | 측정 방법 |
|---|---|---|
| 특정 화면 명세 탐색 소요 시간 | 현재 대비 **70% 감소** | 사용자 인터뷰 |
| 기획 파일 개별 전달 건수 | 산출물 파일 전송 **90% 감소** | 팀 설문 |
| 팀 내 주간 활성 사용자 | 팀원 **80% 이상** 주 1회 이상 사용 | 파일 전달 로그 추적 |
| 사용자 만족도 (NPS) | **NPS 40 이상** | 분기별 NPS 설문 |

## 2. 사용자 문제 정의

### 2.1 대상 사용자 (Target Users)

| 페르소나 | 주요 Pain Point | Jobs-to-be-Done |
|---|---|---|
| **주원** (서비스 기획자, 3년차) | MD·HTML 파일을 폴더에 분산 저장. 팀원 요청 시마다 파일 찾아 전달. 최신본 관리가 어려움 | 산출물 파일 하나로 묶어서 링크나 파일 하나만 주고 싶다 |
| **민지** (UI/UX 디자이너, 2년차) | 기획자에게 매번 최신 파일 요청. 어떤 MD가 어떤 화면인지 파악 어려움. 설치·서버 없이 보고 싶음 | 파일 하나 받아서 화면 ID 치면 명세+프로토타입 바로 보고 싶다 |
| **태영** (개발자, 5년차) | 요구사항 문서가 여러 파일에 흩어짐. 개발 중 빠른 명세 확인이 어려움 | 개발 중 화면 ID로 빠르게 명세 검색해서 확인하고 싶다 |

### 2.2 핵심 문제

1. **산출물 파일 분산** — MD·HTML 파일이 로컬·메신저에 분산되어 최신본 파악 불가
2. **컨텍스트 단절** — 문서 명세와 HTML 프로토타입이 연결되지 않아 화면 이해에 시간 소요
3. **공유 비효율** — 팀원 공유 시 여러 파일을 개별 전달해야 하고 버전 혼란 발생
4. **진입 장벽** — 서버 배포 없이 누구나 바로 열람할 수 있는 방법이 없음

## 3. 제품 범위

### 3.1 제품 구성

Spec Viewer는 두 가지 컴포넌트로 구성된다.

| 컴포넌트 | 설명 |
|---|---|
| **① 패키저 (Packager)** | 기획자가 MD·HTML 파일을 투입하면 단일 Viewer HTML 파일을 생성하는 빌드 도구. GitHub Pages로 배포되는 웹 UI 형태로 제공 (Node.js 불필요) |
| **② 뷰어 (Viewer)** | 패키저가 생성한 단일 HTML 파일. GitHub 레포지터리 `releases/` 폴더에 커밋되며 팀원이 직접 다운로드하여 브라우저에서 열면 즉시 실행 |

### 3.2 In-Scope (v1.0)

- **① 패키저**: MD 문서 파싱·HTML 변환, HTML 프로토타입 인라인 임베딩, 단일 HTML 파일 출력
- **② 뷰어**: MD 렌더링, 목차 사이드바, 화면 ID 탐색, 전체 텍스트 검색
- **② 뷰어**: HTML 프로토타입 iframe 실행, 디바이스 프리뷰 (Mobile / Tablet / Desktop)
- **② 뷰어**: 화면 ID 기반 자동 매핑 + 스플릿 뷰 (문서 + 프로토타입 나란히 보기)
- **② 뷰어**: 즐겨찾기·최근 열람 (브라우저 LocalStorage 기반)
- **② 뷰어**: 섹션·화면 ID 기반 코멘트 작성·조회·패널 (LocalStorage 저장, JSON 내보내기/가져오기)
- 완전 오프라인 동작 (외부 CDN 의존 없음)
- GitHub 레포지터리 관리: 소스코드 버전 관리, `releases/` 폴더에 Viewer HTML 배포, GitHub Pages로 패키저 웹 UI 제공

### 3.3 Out-of-Scope (추후 버전 검토)

| 기능 | 검토 시점 |
|---|---|
| 서버 배포 및 사용자 인증 기능 | v2+ — 팀 규모 확대 시 검토 |
| 실시간 공동 편집 기능 | v2+ |
| Figma·Notion 등 외부 툴 연동 | v3+ |
| 버전 관리 이력 UI | v2+ |

## 4. 기능 상세

### 4.1 ① 패키저 (Packager)

#### 4.1.1 입력 및 처리

- **입력**: MD 파일들 + HTML 프로토타입 파일들 + 매핑 설정 파일 (선택)
- **MD 파일**: `marked.js` 등으로 HTML 변환 → JSON 직렬화하여 단일 파일에 인라인 삽입
- **HTML 프로토타입**: Blob URL 또는 `srcdoc` 방식으로 iframe 내 실행 가능하게 인라인 임베딩
- **외부 라이브러리** (스타일·스크립트): 모두 인라인 번들링 (CDN 참조 금지)
- **출력**: 단일 `.html` 파일 (예: `SpecViewer_v1.0_20260518.html`)

#### 4.1.2 화면 ID 자동 매핑

- MD 파일 헤딩에서 `SCR-NNN` 패턴 자동 추출 → 화면 ID 인덱스 생성
- HTML 파일명에서 `SCR-NNN` 패턴 추출하여 자동 매핑 (예: `SCR-001.html` → `SCR-001`)
- 매핑 설정 파일 (JSON)을 통한 수동 매핑 지원 (자동 매핑 실패 시 보완)

#### 4.1.3 제공 방식 및 레포지터리 구조

| 구성 요소 | 설명 |
|---|---|
| **패키저 웹 UI** | GitHub Pages (`docs/index.html`)로 배포. 팀원은 URL 접속 → 파일 드래그앤드롭 → Viewer HTML 즉시 다운로드. Node.js 설치 불필요 |
| **소스코드 관리** | `src/` 폴더에 패키저·뷰어 소스코드 위치. GitHub로 버전 관리, PR 기반 기능 개발 |
| **Viewer 배포** | 기획자가 빌드한 Viewer HTML을 `releases/` 폴더에 커밋. 팀원은 레포에서 직접 파일 다운로드 |

**레포지터리 디렉터리 구조 (권장)**

```
src/          — 패키저·뷰어 소스코드
docs/         — GitHub Pages 루트 (패키저 웹 UI)
  index.html  — 패키저 웹 UI 진입점
releases/     — 빌드된 Viewer HTML 파일 (팀원 다운로드 대상)
README.md     — 사용 방법, 레포 구조 안내
```

### 4.2 ② 뷰어 (Viewer)

#### 4.2.1 레이아웃 구조

| 영역 | 내용 |
|---|---|
| **좌측 사이드바** | 파일 목록 + 선택 문서의 계층형 목차 |
| **중앙 메인** | 문서 뷰어 또는 스플릿 뷰 (문서 + 프로토타입) |
| **상단 툴바** | 글로벌 검색, 뷰 전환 (문서/프로토타입/스플릿), 디바이스 프리뷰 선택 |

#### 4.2.2 문서 뷰어

- GitHub Flavored Markdown (GFM) 기준 렌더링
- 코드 블록 신택스 하이라이팅 (인라인 번들 포함)
- 표, 이미지, 내부 앵커 링크 렌더링
- 목차 사이드바: H1~H4 자동 추출, 활성 섹션 하이라이트, 클릭 스무스 스크롤

#### 4.2.3 프로토타입 뷰어

- HTML 프로토타입을 `srcdoc` 방식 iframe으로 실행
- 디바이스 프리뷰 프리셋: Mobile (375×812), Tablet (768×1024), Desktop (1440×900), Custom
- **스플릿 뷰**: 좌측 문서 명세 + 우측 프로토타입 iframe 동시 표시, 분할 비율 조절 가능

#### 4.2.4 검색 및 탐색

- **글로벌 검색**: 모든 문서 대상 키워드 검색, 결과 하이라이트
- **화면 ID 검색**: `SCR-NNN` 입력 시 해당 문서 섹션 + 연결 프로토타입 즉시 이동
- **즐겨찾기**: 자주 확인하는 화면 ID 북마크 (브라우저 LocalStorage 저장)
- **최근 열람**: 현재 세션 내 열람 이력 표시

#### 4.2.5 코멘트

- 문서 섹션 또는 화면 ID 단위로 코멘트 작성·수정·삭제
- 작성자 이름 (사용자 입력) + 작성 일시 자동 기록
- **코멘트 패널**: 전체 코멘트를 문서·화면 ID 기준으로 정렬하여 모아보기
- 코멘트 클릭 시 해당 섹션으로 즉시 이동
- **저장 방식**: 브라우저 LocalStorage — 동일 기기에서 재열람 시 유지
- **JSON 내보내기/가져오기** — 코멘트 파일을 팀원과 별도 공유 가능
- ※ 단일 HTML 특성상 다기기 실시간 공유는 미지원. JSON 파일 교환 방식으로 협업 보완.

## 5. 화면 구조 (IA)

### 5.1 주요 화면 목록

| 화면 ID | 화면명 | 설명 |
|---|---|---|
| **PKG-001** | 패키저 — 파일 등록 | [패키저] MD·HTML 파일 드래그앤드롭 등록, 유형 지정, 매핑 설정 |
| **PKG-002** | 패키저 — 빌드 결과 | [패키저] 빌드 진행 상태, 완료 후 단일 HTML 파일 다운로드 |
| **SCR-001** | 뷰어 — 홈 | [뷰어] 최근 열람·즐겨찾기, 전체 파일 목록 요약 |
| **SCR-002** | 뷰어 — 문서 뷰어 | [뷰어] MD 렌더링, 목차 사이드바, 전체 텍스트 검색, 코멘트 작성 |
| **SCR-003** | 뷰어 — 프로토타입 뷰어 | [뷰어] HTML iframe, 디바이스 프리뷰 전환 |
| **SCR-004** | 뷰어 — 스플릿 뷰 | [뷰어] 문서 + 프로토타입 나란히 보기, 분할 비율 조절 |
| **SCR-005** | 뷰어 — 검색 결과 | [뷰어] 글로벌 검색·화면 ID 검색 결과, 하이라이트 |
| **SCR-006** | 뷰어 — 코멘트 패널 | [뷰어] 전체 코멘트 목록, 섹션 이동, JSON 내보내기/가져오기 |

### 5.2 핵심 사용자 흐름

**흐름 1 — 기획자의 패키징 및 공유**

1. 패키저 실행 → MD 파일·HTML 파일 등록
2. 화면 ID 자동 매핑 결과 확인 및 필요 시 수동 매핑 조정
3. 빌드 실행 → 단일 HTML 파일 다운로드
4. 메신저·드라이브 등으로 팀원에게 파일 공유

**흐름 2 — 팀원의 화면 명세 확인**

1. 공유받은 HTML 파일을 브라우저에서 열기 (설치·로그인 불필요)
2. 상단 검색창에 화면 ID 입력 (예: `SCR-010`)
3. 문서 뷰어 (SCR-002)에서 해당 섹션 바로 이동
4. 스플릿 뷰 (SCR-004) 전환 → 명세 + 프로토타입 동시 확인

**흐름 3 — 팀원의 코멘트 작성 및 공유**

1. 문서 뷰어에서 코멘트를 남기고 싶은 섹션 선택
2. 코멘트 입력창에 작성자 이름 + 내용 작성 후 저장 (LocalStorage 저장)
3. 코멘트 패널 (SCR-006)에서 전체 코멘트 확인
4. 코멘트 JSON 내보내기 → 메신저·드라이브로 팀원에게 전달
5. 수신한 팀원은 코멘트 가져오기(Import)로 동일 코멘트 반영

## 6. 기술 요구사항

### 6.1 아키텍처 원칙

| 원칙 | 내용 |
|---|---|
| **단일 HTML 파일 완결** | 모든 스타일·스크립트·데이터를 인라인으로 번들링 |
| **외부 의존성 Zero** | CDN·외부 API 참조 금지. 오프라인 환경 완전 동작 보장 |
| **순수 클라이언트 사이드** | 서버 사이드 로직 없음. 브라우저 JS만으로 동작 |
| **LocalStorage 활용** | 즐겨찾기·코멘트 등 영속 상태는 브라우저 LocalStorage에 저장 |
| **GitHub 중심 관리** | 소스코드·버전 이력은 GitHub, 패키저 UI는 GitHub Pages, Viewer 파일은 `releases/` 폴더로 일원화 |

### 6.2 기술 스택 (권장)

| 컴포넌트 | 기술 스택 |
|---|---|
| **패키저** | 순수 HTML + Vanilla JS (브라우저에서 직접 실행, Node.js 불필요). GitHub Pages (`docs/`)로 배포 |
| **뷰어 UI** | 순수 HTML + CSS + Vanilla JS (프레임워크 최소화로 번들 크기 절감) |
| **MD 렌더링** | `marked.js` (인라인 번들) |
| **코드 하이라이팅** | `highlight.js` (인라인 번들, 필요 언어만 선택 번들링) |
| **프로토타입 임베딩** | iframe `srcdoc` 방식 또는 Blob URL (브라우저 보안 정책 고려) |
| **검색 인덱싱** | 패키저 (브라우저 JS)에서 전처리로 검색 인덱스 생성 후 JSON으로 인라인 삽입 |
| **버전 관리·배포** | GitHub 레포지터리. 소스: `src/`, 패키저 UI: `docs/` (GitHub Pages), Viewer 배포: `releases/` |

### 6.3 파일 크기 관리

| 기준 | 목표 크기 |
|---|---|
| 패키징된 단일 HTML 파일 (프로토타입 포함) | **50MB 이하** |
| MD 문서만 포함 시 | **5MB 이하** 목표 |
| HTML 프로토타입이 대용량일 경우 | 파일 크기 경고 및 압축 권고 |

## 8. 가정 및 리스크

### 8.1 가정 (Assumptions)

- 팀원은 최신 웹 브라우저가 설치된 환경에서 HTML 파일을 열람한다
- 팀 전원이 GitHub 레포지터리에 접근 권한을 가지고 있다
- 기획자가 작성하는 화면설계서 MD 파일 내 화면 ID는 일관된 형식 (`SCR-NNN`)으로 관리된다
- HTML 프로토타입은 단일 HTML 파일이거나 외부 리소스가 최소화된 형태이다

### 8.2 리스크 및 대응 방안

| 번호 | 리스크 | 발생 가능성 | 대응 방안 |
|---|---|---|---|
| R-01 | 화면 ID 형식 비일관성으로 자동 매핑 실패율 높음 | 높음 | 수동 매핑 JSON 설정 파일 지원, 화면 ID 가이드 문서 배포 |
| R-02 | HTML 프로토타입 인라인 임베딩 시 브라우저 보안 정책 (CSP 등) 충돌 | 중간 | `srcdoc`·Blob URL 방식 병행 테스트, 크로스브라우저 검증 강화 |
| R-03 | 대용량 프로토타입 포함 시 단일 HTML 파일 크기 과도 증가 | 중간 | 파일 크기 경고 기능, 이미지 압축 옵션 제공 |
| R-04 | GitHub 레포 `releases/` 폴더에 대용량 Viewer HTML 파일이 누적되어 레포 크기 증가 | 낮음 | 파일 크기 50MB 이하 권고, 오래된 릴리즈는 GitHub Releases (태그)로 이관 |
| R-05 | LocalStorage 기반 상태가 파일 재배포 시 초기화됨 | 낮음 | 즐겨찾기 내보내기·가져오기 기능 제공 검토 |
